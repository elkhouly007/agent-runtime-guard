# Skill: Content Engine

## Trigger

Use when systematically producing content across multiple channels (blog posts, LinkedIn, Twitter/X, newsletters, documentation articles) from a single source or topic — especially for teams that need consistent output without manual repetition for each format.

## Process

### Phase 1 — Source Extraction

1. Identify the core idea or input: a product launch, a technical insight, a case study, a feature, a lesson learned.
2. Extract the key points:
   - Core claim or thesis (1 sentence)
   - 3-5 supporting points or evidence
   - Target audience segment (developers, founders, buyers, general public)
   - Desired action (read more, sign up, share, contact sales)
3. If the source is a long document, article, or transcript — summarize it into these 4 elements before proceeding to Phase 2.

### Phase 2 — Content Adaptation by Channel

From a single source, produce channel-specific variants:

| Channel | Format | Length | Tone |
|---|---|---|---|
| Blog post | Long-form with H2/H3 structure | 800–1500 words | Authoritative, helpful |
| LinkedIn article | Personal narrative + insight | 400–700 words | Professional, first-person |
| LinkedIn post | Hook + 5–7 short paragraphs | 150–300 words | Direct, conversational |
| Twitter/X thread | Tweet 1 = hook, tweets 2–8 = points, last = CTA | 20–30 words/tweet | Punchy, concrete |
| Newsletter section | Summary + link + context | 100–200 words | Warm, subscriber-first |
| Documentation | Task-based, step-by-step | As needed | Neutral, precise |

### Twitter/X Thread Template

```
1/ [Hook — most surprising or counterintuitive claim]

2/ [Context — why this matters]

3-7/ [One point per tweet — concrete, scannable]

8/ [Synthesis or takeaway]

9/ [CTA — follow / read full post / sign up]
```

### LinkedIn Post Template

```
[Opening hook — 1-2 lines that stop the scroll]

[3-4 short paragraphs expanding the idea]

[Concrete example or data point]

[Closing insight or question to drive comments]

[Optional: link in first comment]
```

### Blog Post Structure

```markdown
# [Primary keyword + benefit in title]

[Opening hook — story, stat, or counterintuitive claim]

## [H2 — First main section]
[Body — 200-300 words]

## [H2 — Second main section]
[Body — 200-300 words]

## [H2 — Conclusion or takeaway]
[Summary + CTA]
```

### Phase 3 — SEO and Distribution (optional)

- Identify 1 primary keyword and 2-3 secondary keywords from the topic.
- Ensure the primary keyword appears in: title, first paragraph, one H2, meta description.
- Suggest 3 distribution channels (subreddits, newsletters, communities) where this content would land well.

```
Primary keyword: [keyword]
Secondary: [kw1], [kw2], [kw3]
Meta description (under 160 chars): [text]
Slug: /blog/[url-friendly-title]

Distribution:
- r/[relevant_subreddit]
- [Newsletter name] — audience: [description]
- [Community/Slack/Discord] — audience: [description]
```

## Output Format

- Each piece labeled with channel name and intended posting date (if known).
- Ready-to-publish text — no placeholders.
- For threads: numbered tweets, character count shown per tweet.
- For blog posts: include suggested title, meta description (under 160 chars), and slug.

## Content Calendar Integration

When producing a full content batch from one source, output in this structure:

```
SOURCE: [topic or link]
DATE RANGE: [week or month]

DAY 1 — Blog post
[full text]

DAY 2 — Twitter/X thread
[full thread]

DAY 3 — LinkedIn post
[full post]

DAY 5 — Newsletter section
[full section]
```

## Constraints

- Do not invent statistics or customer stories — use only provided data.
- Do not repeat the same phrasing across channels — each piece must feel native to its platform.
- Do not stuff keywords in a way that reads unnaturally.
- If the user provides no content source, ask for the topic, audience, and one concrete data point before drafting.
