## 1. الملخص التنفيذي

**ما هو المشروع اليوم:**
نظام حوكمة وقت تشغيل للوكلاء الذكية (Agent Runtime Guard)، يعمل أساسًا كـ hook إضافي فوق Claude Code. الجوهر الحقيقي هو محرك قرار JavaScript مكوَّن من 11 وحدة، يُقيِّم كل أمر (Bash، كتابة ملف، git push) بناءً على خمسة أبعاد: خطورة الأمر، حساسية المسار، حالة الجلسة، السياسة المكتسبة، وسياق المشروع — ثم يُصدر أحد سبعة إجراءات (allow / route / require-tests / require-review / modify / escalate / block). يُدعم هذا بـ 12 hook تشغيلي، 92 fixture اختبارية، و24 خطوة CI.

**ما يسعى أن يصبح:**
منصة حوكمة متعددة بيئات التشغيل (harnesses) تعمل بالتساوي على Claude Code وOpenClaw وOpenCode وClawCode وAntegravity وCodex، مع ذاكرة سياسة حقيقية، استقلالية محدودة قابلة للتدقيق، وتطبيق فعلي لا وثائقي فحسب.

**أهم حقيقة عن نضجه الحالي:**
المشروع قوي جدًا في بُعد واحد (Claude Code + runtime JS core) وضعيف حقيقي في ثلاثة أبعاد: دعم البيئات الأخرى (مدّعى لكن غير منفَّذ)، ربط الوكلاء/القواعد/المهارات بمحرك القرار (مجرد وثائق)، والتغليف والتوزيع. الأرقام في الوثائق الرئيسية لا تتطابق مع الواقع في ثلاث نقاط على الأقل.

## 2. تدقيق الحالة الراهنة

### 2.1 جدول التقييم

| المجال | التقييم | الملاحظة الحاسمة |
|--------|---------|-----------------|
| الرؤية والاستراتيجية | **مقبول — غير مكتمل** | المهمة واضحة، لكن التعريف متعدد الأهداف في آنٍ (أداة Claude Code + منصة متعددة بيئات + مرجع أمان) |
| المعمارية | **مقبول — غير مكتمل** | فصل جيد بين الوحدات runtime، لكن adapters البيئات الأخرى غير موجودة كوحدات حقيقية |
| النواة الوقتية (runtime core) | **قوي** | 11 وحدة JS، قابلة للتدقيق، ECC_STATE_DIR يُحترم، مؤمَّن ضد أعطال الكتابة |
| التطبيق الفعلي للسياسة | **مقبول — غير مكتمل** | `dangerous-command-gate.js` فقط يستدعي `runtime.decide()`؛ بقية الـ hooks (secret-warning، git-push-reminder) تعمل بأنماط خاصة دون محرك القرار |
| الاستقلالية المحدودة / التعلم | **قوي** | auto-allow-once + trajectory routing منفَّذان ومُختبَران |
| عمق الاختبار والتحقق | **قوي** | 92 fixture، 24 خطوة CI، Ubuntu + Windows، اختبار سلوكي حقيقي |
| التغليف / التثبيت / التوزيع | **ضعيف** | لا npm package، التثبيت بنسخ ملفات يدويًا، لا آلية ترقية |
| جودة الوثائق | **مقبول — غير مكتمل** | تناقضات في الأرقام (agents: 49 مدّعى / 50 حقيقي؛ rules: 81 مدّعى / 82 حقيقي؛ skills: 22 مدّعى / 23 حقيقي)؛ IMPROVEMENT_PLAN.md يحتوي على أرقام أساسية متقادمة بشكل كبير (claims 129 skills؛ الواقع 23) |
| الجاهزية التشغيلية | **ضعيف** | لا dashboard، لا مؤشرات false positive/negative، لا مسار ترقية |
| دعم البيئات المتعددة | **ضعيف** | Claude Code: مدعوم بالكامل. OpenCode/OpenClaw: وثائق موجودة، hooks غير مُكيَّفة. Codex/ClawCode/Antegravity: stubs فارغة فقط |

### 2.2 تفاصيل مناطق الضعف الحرجة

**الوكلاء (agents/) والقواعد (rules/) والمهارات (skills/):**
50 وكيل + 82 قاعدة + 23 مهارة، لكن **لا يوجد مسار كود يصل من محرك القرار إلى هذه الملفات**. هي مراجع توثيقية يستشيرها المستخدم يدويًا. هذا ليس خطأ في حد ذاته، لكنه يجب أن يُصاغ بوضوح كما هو — لا كـ "قدرة منفَّذة".

**secret-warning.js:**
يكشف الأسرار بأنماط regex ثم يعترض بشكل ثنائي. لا يستدعي `runtime.decide()` أبدًا. هذا يعني أن قرارات الأسرار خارج حلقة التحكم الكاملة (السياسة المكتسبة، مستوى الجلسة، السياق).

**OpenCode / OpenClaw:**
WIRING_PLAN.md موجود وكامل، لكن hooks الـ JavaScript هي نفس hooks Claude Code. لكل بيئة معمارية hook مختلفة — OpenCode يستخدم نظام commands JSON، وليس hooks JS مباشرة بالضرورة. لم تُختبر الاتصال الفعلي.

## 3. تعريف الحالة النهائية المستهدفة

النجاح يعني تحقيق الحالة التالية بشكل مدعوم بأدلة قابلة للتحقق:

| البُعد | التعريف الملموس |
|--------|----------------|
| **ذكاء قرار وقت التشغيل** | كل أمر يمر عبر `runtime.decide()` — بما في ذلك hooks الأسرار والـ git push — مع سياسة موحدة |
| **استقلالية محدودة** | Auto-allow-once + trajectory routing + project-scoped trust posture يعملون معًا؛ لا قرار auto-allow بدون سجل مدقَّق |
| **قابلية الشرح** | كل قرار يُنتج `explanation` + `decisionSource` + `trajectoryNudge` قابلًا للقراءة في JSONL |
| **تطبيق حقيقي لا وثائقي** | في وضع ENFORCE، `process.exit(2)` لا يصل إليه إلا بعد `console.error()` دائمًا؛ لا مسار يتجاوز القرار صامتًا |
| **توافق متعدد البيئات** | adapters حقيقية لكل بيئة مدعومة؛ لا harness تدّعي الدعم بدون hook adapter مُختبَر |
| **عزل الحالة per-project** | `ECC_STATE_DIR` يعمل لجميع الوحدات بما فيها hooks-level log |
| **الرصد / الرؤية** | `ecc-cli.sh log` و`ecc-cli.sh status` يعملان ويعرضان بيانات حقيقية |
| **التثبيت والترقية** | script تثبيت واحد يعمل على Mac/Linux/Windows؛ مسار ترقية موثق ومختبر |
| **جودة الإصدار** | لا يخرج إصدار بدون: CI أخضر، CHANGELOG مُحدَّث، أرقام مُتحقق منها في الوثائق |

## 4. الأهداف وغير الأهداف

### الأهداف الرئيسية

1. **تطبيق فعلي للسياسة** — كل hook يمر عبر `runtime.decide()` دون استثناء.
2. **دعم بيئتين إضافيتين بشكل حقيقي** — OpenClaw أو OpenCode مع adapters مُختبَرة.
3. **عزل الحالة الكامل** — جميع ملفات I/O تمر عبر `ECC_STATE_DIR` دون استثناء.
4. **قابلية التثبيت الموثوقة** — script واحد، لا نسخ يدوي.
5. **صدق الوثائق** — لا رقم في الوثائق يتعارض مع الواقع.

### الأهداف الثانوية

6. قراءة الوكلاء/القواعد ديناميكيًا ضمن سياق القرار (اختياري، مبرَّر).
7. مقاييس false positive/negative في سياق حقيقي.
8. آلية ترقية للمستخدمين الحاليين.

### غير الأهداف (صريحة)

- **ليس هدفًا:** دعم Codex/ClawCode/Antegravity قبل اكتمال Claude Code + harness واحد إضافي.
- **ليس هدفًا:** بناء LLM أو استدعاء API خارجي في مسار القرار.
- **ليس هدفًا:** استبدال نظام الـ hooks الخاص بكل بيئة — الهدف adapter، لا استبدال.
- **ليس هدفًا:** توحيد الوكلاء/القواعد/المهارات ضمن محرك القرار الآن (يُؤجَّل).
- **ليس هدفًا:** نشر npm package عام في هذه المرحلة.
- **ليس هدفًا:** المزيد من fixtures العددية بدون فجوات حقيقية في التغطية.

### 5.1 جدول مستويات الدعم

| البيئة | مستوى الدعم | الحالة الحالية | ما هو موجود | ما ينقص | افتراضات وقت التشغيل المختلفة | الأولوية |
|--------|------------|--------------|------------|--------|-------------------------------|---------|
| **Claude Code** | أولي كامل (first-class) | ✅ مدعوم بالكامل | 12 hook JS، WIRING_PLAN، examples، 92 fixture، CI كامل | لا شيء جوهري — هناك تحسينات لكن لا فجوات | hooks JS مباشرة في settings.json | **1** |

| **OpenCode** | مبني على adapter | ⚠️ جزئي | opencode.safe.jsonc، WIRING_PLAN، examples/ | Hook adapter، اختبار، fixture set | يستخدم commands JSON مختلف عن hooks JS | **3** |
| **Codex** | مؤجَّل | 🔴 stub فارغ | COMPATIBILITY_NOTES.md فحسب | كل شيء | غير معروف — يحتاج بحث معمارية | **مؤجَّل** |
| **ClawCode** | مؤجَّل | 🔴 stub فارغ | COMPATIBILITY_NOTES.md فحسب | كل شيء | غير معروف | **مؤجَّل** |
| **Antegravity** | مؤجَّل | 🔴 stub فارغ | COMPATIBILITY_NOTES.md فحسب | كل شيء | غير معروف | **مؤجَّل** |

### 5.2 التوصية الاستراتيجية

**دعم الكل بالتساوي فكرة سيئة.** السبب: adapters البيئات تتطلب فهمًا عميقًا لمعمارية hook كل بيئة قبل أي تطوير. الادعاء بالدعم بدون adapter مُختبَر هو كذب ناعم يضر بمصداقية المشروع.

التوصية: **نموذج ثلاثي الطبقات:**
- **الطبقة 1 (الآن):** Claude Code فقط — أولي كامل، محسَّن، مُختبَر.
- **الطبقة 2 (السبرينت التالي):** OpenClaw أو OpenCode واحدة فحسب — adapter حقيقي بدون ادعاء الثانية.
- **الطبقة 3 (مستقبلي):** Codex/ClawCode/Antegravity — لا تُدرَج في harness matrix حتى يوجد كود فعلي.

### 5.3 ما يلزم لكل adapter قبل ادعاء الدعم

```
□ فهم معمارية hooks الخاصة بالبيئة (كيف تُمرَّر المعطيات، ما شكل stdin/stdout)
□ adapter يُحوِّل صيغة البيئة إلى صيغة runtime.decide() input
□ fixture set خاص بالبيئة (حد أدنى 10 fixtures)
□ check script يُشغِّل adapter باستخدام ECC_STATE_DIR مؤقت
□ wiring doc يصف خطوات التثبيت الفعلية
□ CI step يشغّل الاختبار
```

### 6.2 ما يجب أن ينتقل إلى adapters البيئات

```
harnesses/
├── claude-code/
│   ├── hooks/             ← hooks JS الحالية (12 ملف)
│   ├── wiring.md          ← موجود بالفعل
│   └── adapter.js         ← يستدعي runtime.decide() من hooks
├── openclaw/
│   ├── adapter.js         ← جديد: يُحوِّل openclaw hook input → runtime input
│   ├── wiring.md          ← موجود، يُحدَّث
│   └── fixtures/          ← جديد
└── opencode/
    ├── adapter.js         ← جديد
    ├── opencode.safe.jsonc ← موجود
    └── fixtures/          ← جديد
```

**المبدأ:** النواة لا تعرف البيئة. البيئة لا تعرف التفاصيل الداخلية للنواة. الـ adapter هو الجسر الوحيد.

### 6.3 ما يجب في السياسة / الإعداد

- `ecc.config.json` يبقى على مستوى المشروع (ليس على مستوى البيئة).
- `ECC_STATE_DIR` يُعيِّن مسار جميع الملفات — بدون استثناء. **hook-utils.js يجب أن يُصلح** ليستخدم `ECC_STATE_DIR` بدلًا من المسار المُشفَّر `~/.openclaw/ecc-safe-plus/`.
- إضافة `ECC_HARNESS=claude-code|openclaw|opencode` كمتغير بيئة يُعدِّل سلوك adapter.

### 6.4 ما يجب أن يكون توليديًا لا يدويًا

| المادة | الوضع الحالي | التوصية |
|--------|-------------|---------|
| `artifacts/status/status-summary.txt` | يُولَّد بـ generate-status-artifact.sh | **يبقى توليديًا** |
| `references/parity-report.md` | يُولَّد بـ generate-parity-report.sh | **يبقى توليديًا** |
| `references/superiority-evidence.md` | يُولَّد | **يبقى توليديًا** |
| أرقام الوكلاء/القواعد/المهارات في الوثائق | يدوي حاليًا (تناقضات) | **يُولَّد** من فحص الأدلة |
| `IMPROVEMENT_PLAN.md` أرقام الأساس | يدوي متقادم | **يُصلح** أو يُولَّد |

### 6.5 أين تعيش الحالة

```
~/.openclaw/agent-runtime-guard/   ← حالة runtime (policy, session, journal)
$ECC_STATE_DIR/                    ← override كامل لجميع الوحدات
$PROJECT_ROOT/.ecc/                ← حالة per-project (مقترح جديد — يمنع تسرب الحالة بين مشاريع)
```

**مشكلة حالية:** حالة `hook-utils.js` تكتب في `~/.openclaw/ecc-safe-plus/` مُشفَّرة. هذه مخالفة للعزل. يجب أن تُصلَح.

## 8. خارطة الطريق الكاملة

### المرحلة 1: إصلاح الصدق الداخلي (sprint واحد، صغير)

**الهدف:** لا يوجد ادعاء بدون دليل. أرقام صحيحة. مسارات مُشفَّرة مُصلَحة.

**لماذا تأتي أولًا:** المشروع لا يستطيع التقدم بمصداقية مع تناقضات في الوثائق ومسار I/O مكسور في hook-utils.

**النتائج المتوقعة:**
- IMPROVEMENT_PLAN.md أرقام مُصحَّحة أو مُحذوفة
- README أرقام دقيقة (تُولَّد تلقائيًا)
- hook-utils.js يحترم ECC_STATE_DIR لجميع كتاباته
- check script يتحقق من صدق جميع الأرقام المُعلَنة

**الملفات المتأثرة:**
- `claude/hooks/hook-utils.js` (إصلاح المسار)
- `IMPROVEMENT_PLAN.md` (تصحيح/حذف الأرقام المتقادمة)
- `README.md` (توليد الأرقام أو تصحيحها)
- `scripts/check-runtime-core.sh` (T2.4 ECC_STATE_DIR test)

**معايير القبول:**
- `grep -r "ECC_SAFE_PLUS\|ecc-safe-plus" claude/hooks/` → لا نتائج
- `bash scripts/check-hook-edge-cases.sh` مع `ECC_STATE_DIR=$tmp` → لا ملفات تُكتب خارج `$tmp`
- أرقام README تطابق `find agents/ rules/ skills/ -name "*.md" | wc -l`

### المرحلة 2: توحيد مسار القرار (sprint متوسط)

**الهدف:** كل hook مهم يمر عبر `runtime.decide()`.

**لماذا تأتي ثانيًا:** السبب المعلَن للمشروع هو حوكمة موحدة. إذا كان secret-warning يعمل بمنطق مستقل، فمحرك القرار ليس الحاكم الفعلي.

**النتائج المتوقعة:**
- `secret-warning.js` يستدعي `runtime.decide()` مع `tool: 'SecretScan', payloadClass: 'secret'`
- `git-push-reminder.js` يستدعي `runtime.decide()` مع `tool: 'Bash', command: <detected push command>`
- السياسة المكتسبة تعمل لقرارات الأسرار أيضًا
- 10 fixtures جديدة تختبر المسار الجديد

**الملفات المتأثرة:**
- `claude/hooks/secret-warning.js`
- `claude/hooks/git-push-reminder.js`
- `tests/fixtures/hooks/` (fixtures جديدة)
- `scripts/check-hook-edge-cases.sh` (cases جديدة)

**المخاطر:**
- secret-warning قد يتصرف بشكل مختلف إذا قررت السياسة المكتسبة السماح بسرّ ما — هذا مقصود لكن يحتاج اختبارًا دقيقًا للتأكد من أنه لا يُلغي أنماط السرّ الحرجة

**معايير القبول:**
- `grep "runtimeDecision\|decide(" claude/hooks/secret-warning.js` → موجود
- CI أخضر لجميع الـ fixtures الجديدة
- قرار BLOCK لأسرار حرجة لا يتأثر بالسياسة المكتسبة (تُضاف استثناءات صريحة إن لزم)

## 13. السبرينت التالي الأمثل

### الهدف: إصلاح الصدق الداخلي + ECC_STATE_DIR في hook-utils

**لماذا يأتي الآن:**
الانتقال من "نظام Claude Code متين" إلى "نظام متعدد البيئات ذو مصداقية" يبدأ بهذه الخطوة. لا يُبنى على أساس متصدع — والمسار المُشفَّر في hook-utils + التناقضات في الوثائق هما أساس متصدع.

**تعريف "منتهٍ":**

```
□ hook-utils.js: logDir = ECC_STATE_DIR || path.join(os.homedir(), '.openclaw/ecc-safe-plus')
□ check-hook-edge-cases.sh: ECC_STATE_DIR=$tmp → لا كتابات خارج $tmp
□ README أرقام agents/rules/skills مُتحقَّق منها بـ check script
□ IMPROVEMENT_PLAN.md الأرقام المتقادمة محذوفة أو مُصحَّحة
□ هذا الملف (unified-master-plan.md) موجود ومُراجَع
□ CI أخضر بعد التعديلات
```

**الأدلة المطلوبة في النهاية:**
- `bash scripts/check-hook-edge-cases.sh` مع `ECC_STATE_DIR=$(mktemp -d)` → أخضر + لا ملفات خارج `$TMPDIR`
- CI أخضر على ubuntu + windows

## 14. الحكم النهائي الصريح

### ما يجيده المشروع فعلًا

- **نواة القرار runtime** هي من أفضل ما يُبنى في هذا النوع من الأدوات: وحدات مفصولة، قابلة للاختبار، مؤمَّنة ضد أعطال I/O، تحترم ECC_STATE_DIR، مع session trajectory و learned policy حقيقيَّين.
- **CI والـ fixtures** نموذجيان: 92 fixture، 24 خطوة، Ubuntu + Windows، اختبار سلوكي لا وجودي فحسب.
- **الصدق في SECURITY_MODEL** مُقدَّر: يُعترف صراحةً بالقيود (TOCTOU، prompt injection هيوريستي، obfuscation bypass).

### ما يدَّعيه أكثر مما هو عليه

- **"دعم متعدد البيئات"** — Claude Code فقط له دعم حقيقي. الادعاء بـ OpenClaw/OpenCode كـ "supported" بدون adapter مُختبَر هو تسويق لا واقع.
- **"نظام وكلاء/قواعد/مهارات"** — هي مراجع توثيقية، ليست قدرات runtime.
- **أرقام الوثائق** — ثلاث تناقضات معروفة وواحدة صارخة (129 skills في خطة التحسين / الواقع 23).

### ما يجب أن يحدث كي يصبح ممتازًا

1. توحيد مسار القرار — كل hook عبر `runtime.decide()`.
2. adapter حقيقي مُختبَر لبيئة ثانية.
3. قياس false positive/negative على corpus حقيقي.
4. مسار تثبيت وترقية بدون نسخ يدوي.
5. توقف عن إضافة features قبل إصلاح ما هو معطوب.

### ما يجب إيقافه فورًا

- ❌ إضافة stubs جديدة لبيئات (Codex، ClawCode، Antegravity) بدون كود.
- ❌ تحديث أرقام الوثائق يدويًا — يجب أن تُولَّد.
- ❌ ادعاء دعم بيئة دون adapter + CI step + fixture set.

### ما يجب مضاعفته

- ✅ runtime core: قوي، يستحق مزيدًا من التغطية الاختبارية.
- ✅ CI discipline: نموذجي، يُوسَّع لكل adapter جديد.
